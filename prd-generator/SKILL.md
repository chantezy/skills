# PRD 需求文档生成

## 概述

生成全面、结构清晰的产品需求文档（PRD），遵循行业最佳实践。本技能帮助产品经理创建清晰、可操作的需求文档，协调各方利益相关者并指导开发团队。

## 核心工作流程

当用户请求创建 PRD 时（例如："为用户认证功能创建 PRD"、"帮我写一个PRD"），请遵循以下工作流程：

### 步骤 1：收集背景信息

在生成 PRD 之前，通过发现式对话收集必要信息：

**必要信息：**
- **功能/产品名称**：我们要构建什么？
- **问题陈述**：这能解决什么问题？
- **目标用户**：这是给谁用的？
- **业务目标**：我们想要实现什么？
- **成功指标**：我们将如何衡量成功？

**需要询问的问题：**

```
1. 您想要解决什么问题？
2. 这个功能的主要用户/受众是谁？
3. 关键的业务目标是什么？
4. 有什么技术限制我们需要了解的吗？
5. 成功是什么样的？如何衡量？
```

**注意：** 如果用户提前提供了详细的简报或需求，您可以跳过一些问题。始终对缺失的关键信息进行澄清。

### 步骤 2：生成 PRD 结构

PRD 包括以下部分，如果不需要的内容可以省略，如果有好的表述方式也可以自己规划，但要注意不要重复内容进行描述，要尽量简洁通俗易懂：

1. **一、需求背景** - 需要解决的问题及背景，问题的清晰表述，我们为谁处理，要做到什么程度
2. **二、功能范围** - 功能影响范围及衍生情况的考虑，范围内和范围外的内容
3. **三、业务流程** - 用户、功能流程简述
4. **四、需求详情** - 详细的功能需求
5. **五、操作流程** - 用户操作路径
6. **六、相关需求链接** - （用户自己填写）

### 步骤 3：输出 PRD

将收集的信息按照结构模板生成 Markdown 格式的 PRD 文档。

## PRD 模板

使用 `references/prd-template.md` 作为基础模板，根据实际需求调整内容。

## 最佳实践

### 编写高质量需求

**好的需求应该：**
- **具体**：清晰明确
- **可衡量**：可以验证/测试
- **可实现**：技术可行
- **相关**：与用户/业务价值相关

**避免：**
- 模糊语言（"快速"、"简单"、"直观"）
- 实施细节（让程序员决定如何做）
- 范围蔓延（坚持核心需求）
- 未经验证的假设

### 范围管理

**范围内部分：**
- 列出包含的特定功能/能力
- 明确且详细

**范围外部分：**
- 明确说明不包括什么
- 防止范围蔓延
- 管理利益相关者的期望
- 可以包含"未来考虑"

## 使用模式

### 模式 1：新功能 PRD

**用户请求：** "为我们的移动应用添加深色模式创建 PRD" 或 "帮我写一个深色模式的需求文档"

**执行：**
1. 询问关于深色模式需求的发现式问题
2. 使用模板生成 PRD

### 模式 2：功能优化 PRD

**用户请求：** "编写改进我们搜索功能的需求"

**执行：**
1. 收集当前搜索限制的背景信息
2. 识别用户痛点和期望的改进
3. 生成聚焦于以下内容的 PRD：
   - 当前状态分析
   - 拟议的增强功能
   - 影响评估

## 验证清单

在最终确定 PRD 之前，验证：

- [ ] **问题清晰**：任何人都能理解我们要解决什么
- [ ] **用户已确定**：我们知道这是给谁的
- [ ] **成功可衡量**：我们能判断它是否有效
- [ ] **范围有界限**：明确范围内外
