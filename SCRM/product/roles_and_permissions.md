# 角色权限体系

## 6 种角色

| 角色 | 功能权限 | 数据权限 | 主要职责 |
|---|---|---|---|
| 超级管理员 | 全部 | 全部 | 系统配置、用户管理、数据安全 |
| 管理员 | 除系统配置外 | 全部 | 业务管理、数据监控、团队协调 |
| 运营经理 | 客户管理 + 运营 + 分析 | 部门 | 运营策略、团队管理、效果评估 |
| 运营专员 | 客户管理 + 运营 | 个人 | 客户维护、活动执行、内容发布 |
| 销售经理 | 客户管理 + 分析 | 部门 | 销售管理、业绩追踪、团队培训 |
| 销售专员 | 客户管理 | 个人 | 客户跟进、商机转化、订单处理 |

## 权限模型

```
功能权限（能看什么菜单 / 按钮）
   +
数据权限（能看到哪些客户 / 数据）
   =
最终权限
```

### 功能权限粒度
- 菜单级
- 按钮级（新增 / 编辑 / 删除 / 导出）
- 字段级（敏感字段如手机号脱敏）

### 数据权限粒度
- 全部数据
- 部门数据（含下级）
- 个人数据（仅自己）
- 自定义（按规则）

## 角色与方案设计的关系

设计任何功能时，AI 必须考虑：

| 思考点 | 举例 |
|---|---|
| 给谁用？ | 这个"创作者孵化"功能给运营经理还是专员？ |
| 看什么数据？ | 运营专员能看到全平台创作者吗？还是只看自己负责的？ |
| 操作权限？ | 运营专员能签约吗？还是只能提交审批？ |
| 跨部门协同？ | 这功能涉及销售 + 运营协作吗？ |

## 适合扩展的方向

### 游戏行业角色补充
- 创作者经纪人（专人对接创作者）
- 公会运营官（专人对接公会会长）
- 内容审核员（UGC 内容审核）
- BD 商务（开发商对接）

### 权限增强
- 字段级脱敏（手机号 / 身份证）
- 临时权限（活动期间扩权）
- 审批流（敏感操作需要审批）

## 给 AI 的提示
- 默认所有方案要标"主要使用角色"
- 涉及敏感数据要考虑脱敏 / 审批
- 现有 6 种角色覆盖通用场景，特殊角色要明确指出"需要新增"
