# 数据架构

## 四层架构

```
┌────────────────────────────────────┐
│ 采集层 Source                       │
│ 企微 | 游戏盒APP | 网页端 | API     │
└────────────────────────────────────┘
              ↓
┌────────────────────────────────────┐
│ 处理层 Process                      │
│ 客户归集 | 标签计算 | 行为分析 | 清洗 │
└────────────────────────────────────┘
              ↓
┌────────────────────────────────────┐
│ 存储层 Storage                      │
│ 客户 | 行为 | 运营 | 日志            │
└────────────────────────────────────┘
              ↓
┌────────────────────────────────────┐
│ 应用层 Application                  │
│ 客户管理 | 客户运营 | 数据分析 | 设置 │
└────────────────────────────────────┘
```

## 数据源详解

### 1. 企业微信（B 端通道）
- 客户基础信息（external_userid / 昵称 / 头像）
- 添加来源（活码 / 手动 / 手机号）
- 聊天记录（来自消息存档）
- 标签（企微侧标签）
- 客户群信息

### 2. 游戏盒 APP（C 端主战场，独家优势）
- 用户注册 / 登录行为
- 浏览游戏 / 内容
- 下载 / 启动游戏
- 充值 / 订单
- UGC 发布
- 社交关系（关注 / 公会）

### 3. 网页端
- Web 端浏览行为
- 表单提交

### 4. API 接口
- 与开发商系统对接
- 与广告投放平台对接
- 与电商订单系统对接

## 数据处理关键能力

### 客户归集（OneID）
**核心问题**：同一个人在企微、APP、网页端可能有不同 ID
**解决方案**：通过手机号 / unionid / 设备指纹关联，沉淀为 OneID

### 标签计算
**实时标签**：行为触发即打（点击、下单）
**离线标签**：每日凌晨批量计算（RFM、活跃度等）

### 行为分析
事件流处理，支撑漏斗 / 路径 / 归因分析

### 数据清洗
去重 / 异常值 / 脏数据处理

## 数据存储（建议字段集）

### 客户主表
external_userid, unionid, 游戏盒uid, 手机号, 昵称, 头像, 来源, 添加员工, 添加时间, 角色, 状态…

### 行为表
事件 ID, 客户 ID, 事件类型, 事件属性, 时间戳, 渠道, 设备…

### 运营表
活动 ID, 旅程 ID, 触达记录, 互动记录…

### 日志表
操作日志、登录日志、系统日志

## 数据安全
- 客户敏感信息加密存储
- 数据权限隔离
- 跨境传输管控
- 满足《个人信息保护法》

## 给 AI 的提示
- 任何方案要明确"数据从哪来"
- 游戏盒 APP 数据是核心差异化资产，要充分用
- OneID 是基础，做精细化运营之前必须做好
- 实时 vs 离线选型要谨慎（实时贵但快，离线便宜但有延迟）
